01

Overview

The Wallet Console is the single operational interface for the Param platform. It talks to Wallet Backend (platform ops, reads, SuperApp install, users, orgs, RBAC) and ParamGateway (on-chain writes only — SM definition, Schema definition, document create, document transitions). SyncFactory writes to the database when initiated by ParamGateway — the console never writes directly to MongoDB.

Data flow: Definitions (SM + Schema via ParamGateway · SuperApp + RBAC via Wallet Backend) → Workspace create → SuperApp install → Documents (list via Wallet Backend · create/transition via ParamGateway). All on-chain writes go through ParamGateway → gPRM/NATS → SyncFactory.

📌
Backend Architecture Alignment

Wallet Backend: workspace, SuperApp install, org/user/RBAC, profile, documents (read only). ParamGateway: SM definition, Schema definition, document create, document transitions — nothing else. The console does not expose pipeline management or DLQ UI. SyncFactory: writes onchain/offchain definitions and documents when triggered by ParamGateway.

06b

Team RBAC Matrix

Accessed from the Definitions sidebar — under SuperApp Definitions. The RBAC Matrix defines per-role-team, per-state permissions for each SM linked in a SuperApp. Created via POST /definitions/team-rbac-matrix (Wallet Backend). One matrix per linked SM, required before SuperApp install. Supports Create, View, and Edit — Edit reopens the same matrix form pre-filled.

Team RBAC Matrix — Deployment

The Team RBAC Matrix is required before any SuperApp can be installed. It is created via POST /definitions/team-rbac-matrix (Wallet Backend). One matrix call per linked SM — if a SuperApp has 2 SMs, submit 2 matrix calls. The console auto-parses roles and states from the selected SM definition and generates the grid. Each cell is RW / RO / N/A per Role.Team × State combination.

📌
access key format: "Role.Team"

Each column header in the matrix is RoleName.TeamName — e.g. Consignee.Admin, Consignee.Planners, FF.FF, CHA.CHA. Every role-team pair from the SuperApp definition must have a column. SubState and MicroState rows are indented — they inherit read from the parent state but control write independently.

SubState rows (↳) inherit read from parent state. They only independently control write access. Every state, subState, and microState defined in the SM must have a row — no gaps allowed. Most-specific match wins at runtime: microState > subState > state.
⚠️
Required before install

A SuperApp cannot be installed into a workspace until its Team RBAC Matrix has been saved for every linked SM. The Install button on the SuperApp definition is disabled until all matrices are present. Once installed, the workspace gets its own copy of the matrix that can be customised post-install from Settings → RBAC.

07

Shell & Navigation

Fixed 3-panel layout. Top bar (right side) contains three items: ◇ Definitions pill (navigates to param_definitions, outside workspace context), workspace switcher dropdown (shows active workspace + ▾; clicking opens a popover listing all workspaces + "+ Create Workspace"), and a single role badge. Only one role is ever shown — the logged-in user's real role when operating normally (e.g. Admin ▾), or the impersonated demo role when Demo Mode is active (e.g. 🎭 Buyer). Never both at the same time. Left nav: contextual — shows SuperApp document types when inside an app. Center: main content. Right: detail inspector.

👤
Single role in top bar — always

Normal session: shows the user's real role (e.g. Admin ▾). Demo Mode active: shows the demo role with 🎭 prefix (e.g. 🎭 Buyer). The ▾ on the real role opens a role-switcher if the user has multiple roles. There is no situation where both a real role badge and a demo badge appear simultaneously.

After installing SuperApps into a workspace, the user selects one from the SuperApps list. 📌
Navigation logic

The active workspace is always visible in the top-right switcher. Switching workspace resets the SuperApp selection.

Navigation rules

1. "SuperApps" nav item always goes back to the full SuperApp list for the active workspace.
2. The SuperApp section header in the sidebar (e.g. "🚀 Bosch-Invoice") is the active SuperApp — clicking it also goes back to the list.

10

Document Testing

Documents are scoped to an active SuperApp. Selecting a state from the left sidebar shows documents at that SM state. Each state has horizontal subtabs for its substates — selecting a subtab filters the document list to that exact substate. "All" subtab shows all substates combined. List via Wallet Backend GET /documents. Create/transition via ParamGateway.

🔀
ParamGateway — the console uses it only for these calls

(1) SM definition deploy — POST /pipelines with mnemonic:"define", defType:"statemachine". (2) Schema definition deploy — same, defType:"schema". (3) Document create — POST /pipelines/:id/execute with document payload. (4) Document transition — same execute endpoint with _chain.stateTo override per record. No pipeline management UI, no DLQ, no batch list is exposed anywhere in the console.

Example below uses Shipment Tracker SM (real SM from param_definitions). Two top-level states: Orders (substates: Created, Cancelled) and Shipment (substates: PrePickup → Pickup → PreAlert → HSNPending → Submit → InTransit → Arrived → PoD · terminal: Cancelled). The sidebar shows the two states; clicking a state loads the substate tab bar in the center panel.

Substate tab bar rules

1. Substate tabs are generated dynamically from the SM definition fetched via Wallet Backend — never hardcoded.
2. "All" tab is always the first tab and shows all documents regardless of substate.
3. Active tab is highlighted with an underline accent; selected document's detail panel shows both state and substate (e.g. Shipment · InTransit).
4. If a state has no substates, no tab bar is rendered — the document list fills the full center panel.
5. Transition actions in the right panel reflect what the current substate allows — derived from GET /actions.
