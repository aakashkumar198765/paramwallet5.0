UI Design System v5.0

Param Wallet
Interface Specification

A flexible, AI-generated design system for enterprise SaaS applications. This specification defines the visual language, component architecture, and layout patterns that guide AI-generated client interfaces while ensuring consistency, accessibility, and professional quality across all deployments.

00 — Foundation

Design Philosophy

Param Wallet 5.0 generates completely different user interfaces for each client based on their State Machine definition, Schema structure, and user requirements. This design system provides the visual vocabulary and component language that AI uses to construct client-specific interfaces.

Core Principle

The boilerplate provides capabilities and contracts, not visual decisions. The AI decides layout, navigation, and visual design based on each client's workflow and users. This system ensures consistency without constraining creativity.

01
Workflow-Driven Design
The State Machine defines the UI structure. Navigation, permissions, and data views all flow from SM states and transitions.
02
Data-First Interfaces
Schema defines forms, tables, and validation. The UI renders what the schema describes—no more, no less.
03
Role-Based Visibility
Every element respects SM-defined permissions. Users see only what their role allows—enforced by SMRuntime, not UI logic.
04
Enterprise-Grade Clarity
Dense but readable. Professional color semantics. Clear hierarchy. Built for users who spend hours in the interface daily.
05
Pattern Flexibility
Multiple layout patterns are supported. The AI chooses the right pattern based on workflow complexity, user roles, and data density.

04 — Patterns

Layout Architecture

Param 5.0 supports multiple layout patterns. The AI selects the appropriate pattern based on workflow complexity, user roles, and data density. Each pattern is optimized for specific use cases observed in deployed clients.

Pattern Selection Guide

Sidebar Navigation Default
Vertical sidebar with icon+text navigation. Best for applications with 4-8 primary sections. Supports collapsed mode for power users. Ideal for operational dashboards and data-heavy workflows.
Use when: Multiple SM states, clear workflow stages
Top Navigation Tabs Alternative
Horizontal tab navigation at the top. Best for applications with clearly separated business domains. Cleaner look for executive-facing interfaces. Supports sub-navigation within each tab.
Use when: Domain-separated workflows, executive audience
Three-Panel Layout Advanced
Left sidebar for navigation/entity list, center for primary data, right for details/KPIs. Maximum information density for operational users. Requires wide screens.
Use when: Complex scheduling, drill-down workflows, planners
KPI Dashboard Overview
Summary metrics at top, filterable data table below. Alert banners for time-sensitive issues. Ideal for control towers and monitoring dashboards. Clean, executive-friendly.
Use when: Monitoring, alerts-driven workflows, executives

08 — Theme

Dark and Light Modes

Both themes are first-class citizens with identical component support. The dark theme uses elevated surfaces for depth perception. All semantic colors are theme-aware.

Implementation Rule

No component may reference a raw color value. All colors must flow through CSS custom properties defined in theme token sets. This guarantees theme correctness and enables client brand customization.

Theme Detection

The application should detect system preference on first load and allow manual override. Store preference in localStorage. Apply theme class to <html> element.

09 — AI Generation

AI Generation Rules

These rules govern how the AI constructs client-specific interfaces from the boilerplate. Following these rules ensures consistency while enabling complete customization.

Layout Selection Required
AI must analyze the State Machine to determine layout. SM with 3-5 states → Sidebar navigation. SM with domain-separated states → Top tabs. Complex scheduling workflows → Three-panel layout.
Navigation Derivation Required
Navigation items come from SMRuntime.getNavigationStates(). Never hardcode navigation. Labels come from SM state descriptions. Icons can be inferred from state purpose.
Form Generation Required
Forms are generated from SchemaEngine.getFields(subschema). Field order, types, and validation come from schema. AI decides visual layout (single column, two column, sections).
Table Generation Required
Table columns come from SchemaEngine.getIndexFields(subschema). AI decides which fields to show, column widths, and cell renderers based on field type.

Prohibited Patterns

What AI Must Never Do
  • ✗ Hardcode role names in UI logic — use SMRuntime.canView() and SMRuntime.isOwner()
  • ✗ Fetch SM or Schema at runtime — they are fed as JSON at generation time
  • ✗ Hardcode users or org data — always fetch via sdk.users.getMe() and sdk.users.list()
  • ✗ Make raw HTTP calls — always use SDK methods
  • ✗ Skip validation layers — always run Schema, then validation.js, then businessValidation.js
  • ✗ Expose internal IDs to users — use human-readable identifiers

Supply chain control tower with horizontal tab navigation. KPI summary cards at top, alert banners for exceptions, flat data table with filters. Different sections per business domain (S&OP, Shipment, Customs, Inventory).
